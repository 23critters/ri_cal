Abstract 



   This document specifies a protocol using the iCalendar object
   specification to provide scheduling interoperability between
   different calendaring systems.  This is done without reference to a
   specific transport protocol so as to allow multiple methods of
   communication between systems.  Subsequent documents will define
   profiles of this protocol using specific interoperable methods of
   communications between systems.

   iTIP complements the iCalendar object specification by adding
   semantics for group scheduling methods commonly available in current
   calendaring systems.  These scheduling methods permit two or more

  1. Introduction and Overview 



   This document specifies how calendaring systems use iCalendar
   [I-D.ietf-calsify-rfc2445bis] objects to interoperate with other
   calendaring systems.  In particular, it specifies how to schedule
   events, to-dos, or daily journal entries.  It further specifies how
   to search for available busy time information.  It does so in a
   general way without specifying how communication between different
   systems actually takes place.  Subsequent documents specify transport
   bindings between systems that use this protocol.

   This protocol is based on messages sent from an originator to one or
   more recipients.  For certain types of messages, a recipient may
   reply, in order to update their status and may also return
   transaction/request status information.  The protocol supports the
   ability for the message originator to modify or cancel the original
   message.  The protocol also supports the ability for recipients to
   suggest changes to the originator of a message.  The elements of the
   protocol also define the user roles for its transactions.

   This specification obsoletes RFC 2446 - a list of important changes
   is provided in Appendix A.

2.1. Application Protocol 



   In the iTIP model, an iCalendar object is created and managed by an
   &quot;Organizer&quot;.  The &quot;Organizer&quot; interacts with other CUs by sending one
   or more of the iTIP messages listed above.  &quot;Attendees&quot; use the
   &quot;REPLY&quot; method to communicate their status.  &quot;Attendees&quot; do not make
   direct changes to the master iCalendar object.  They can, however,
   use the &quot;COUNTER&quot; method to suggest changes to the &quot;Organizer&quot;.  In
   any case, the &quot;Organizer&quot; has complete control over the master
   iCalendar object.

2.1.1. Scheduling State 



   There are two distinct states relevant to iCalendar objects used in
   scheduling: the overall state of the iCalendar object and the state
   associated with an &quot;Attendee&quot; in that iCalendar object.

Daboo Expires May 7, 2009 [Page 9]

Internet-Draft iTIP November 2008 



   The state of an iCalendar object is defined by the &quot;STATUS&quot; property
   and is controlled by the &quot;Organizer.&quot;  There is no default value for
   the &quot;STATUS&quot; property.  The &quot;Organizer&quot; sets the &quot;STATUS&quot; property to
   the appropriate value for each iCalendar object.

   The state of a particular &quot;Attendee&quot; relative to a iCalendar object
   used for scheduling is defined by the &quot;PARTSTAT&quot; parameter in the
   &quot;ATTENDEE&quot; property for each &quot;Attendee&quot;.  When an &quot;Organizer&quot; issues
   the initial iCalendar object, &quot;Attendee&quot; status is typically unknown.
   The &quot;Organizer&quot; specifies this by setting the &quot;PARTSTAT&quot; parameter to
   &quot;NEEDS-ACTION&quot;.  Each &quot;Attendee&quot; modifies their &quot;ATTENDEE&quot; property
   &quot;PARTSTAT&quot; parameter to an appropriate value as part of a &quot;REPLY&quot;
   message sent back to the &quot;Organizer&quot;.

2.1.2. Delegation 



   Delegation is defined as the process by which an &quot;Attendee&quot; grants
   another CU (or several CUs) the right to attend on their behalf.  The
   &quot;Organizer&quot; is made aware of this change because the delegating
   &quot;Attendee&quot; informs the &quot;Organizer&quot;.  These steps are detailed in the
   REQUEST method section.

2.1.3. Acting on Behalf of other Calendar Users 



   In many organizations one user will act on behalf of another to
   organize and/or respond to meeting requests.  ITIP provides two
   mechanisms that support these activities.

   First, the &quot;Organizer&quot; is treated as a special entity, separate from
   &quot;Attendees&quot;.  All responses from &quot;Attendees&quot; flow to the &quot;Organizer&quot;,
   making it easy to separate a calendar user organizing a meeting from
   calendar users attending the meeting.  Additionally, iCalendar
   provides descriptive roles for each &quot;Attendee&quot;.  For instance, a role
   of &quot;chair&quot; may be ascribed to one or more &quot;Attendees&quot;.  The &quot;chair&quot;
   and the &quot;Organizer&quot; may or may not be the same calendar user.  This
   maps well to scenarios where an assistant may manage meeting
   logistics for another individual who chairs a meeting.

   Second, a &quot;SENT-BY&quot; parameter may be specified in either the
   &quot;Organizer&quot; or &quot;Attendee&quot; properties.  When specified, the &quot;SENT-BY&quot;
   parameter indicates that the responding CU acted on behalf of the
   specified &quot;Attendee&quot; or &quot;Organizer&quot;.

2.1.5. Message Sequencing 



   CUAs that handle the iTIP application protocol must often correlate a
   component in a calendar store with a component received in the iTIP
   message.  For example, an event may be updated with a later revision
   of the same event.  To accomplish this, a CUA must correlate the
   version of the event already in its calendar store with the version
   sent in the iTIP message.  In addition to this correlation, there are
   several factors that can cause iTIP messages to arrive in an
   unexpected order.  That is, an &quot;Organizer&quot; could receive a reply to
   an earlier revision of a component AFTER receiving a reply to a later
   revision.

   To maximize interoperability and to handle messages that arrive in an
   unexpected order, use the following rules:

   1.  The primary key for referencing a particular iCalendar component
       is the &quot;UID&quot; property value.  To reference an instance of a
       recurring component, the primary key is composed of the &quot;UID&quot; and
       the &quot;RECURRENCE-ID&quot; properties.

   2.  The secondary key for referencing a component is the &quot;SEQUENCE&quot;
       property value.  For components where the &quot;UID&quot; and
       &quot;RECURRENCE-ID&quot; property values are the same, the component with
       the highest numeric value for the &quot;SEQUENCE&quot; property obsoletes
       all other revisions of the component with lower values.

   3.  &quot;Attendees&quot; send &quot;REPLY&quot; messages to the &quot;Organizer&quot;.  For

       property obsoletes all other replies with lower values.

       others.  Similarly, for &quot;Attendee&quot; responses where the &quot;UID&quot;,

       response with the latest &quot;DTSTAMP&quot; overrides all others.

3.2.2.1. Rescheduling an Event 



   The &quot;REQUEST&quot; method may be used to reschedule an event.  A
   rescheduled event involves a change to the existing event in terms of
   its time or recurrence intervals and possibly the location or
   description.  If the recipient CUA of a &quot;REQUEST&quot; method finds that
   the &quot;UID&quot; property value already exists on the calendar, but that the
   &quot;SEQUENCE&quot; (or &quot;DTSTAMP&quot;) property value in the &quot;REQUEST&quot; method is
   greater than the value for the existing event, then the &quot;REQUEST&quot;
   method describes a rescheduling of the event.

3.2.2.2. Updating or Reconfirmation of an Event 



   The &quot;REQUEST&quot; method may be used to update or reconfirm an event.  An
   update to an existing event does not involve changes to the time or
   recurrence intervals, and might not involve a change to the location
   or description for the event.  If the recipient CUA of a &quot;REQUEST&quot;
   method finds that the &quot;UID&quot; property value already exists on the
   calendar and that the &quot;SEQUENCE&quot; property value in the &quot;REQUEST&quot; is
   the same as the value for the existing event, then the &quot;REQUEST&quot;
   method describes an update of the event details, but no rescheduling
   of the event.

   The update &quot;REQUEST&quot; method is the appropriate response to a
   &quot;REFRESH&quot; method sent from an &quot;Attendee&quot; to the &quot;Organizer&quot; of an
   event.

   The &quot;Organizer&quot; of an event may also send unsolicited &quot;REQUEST&quot;
   methods.  The unsolicited &quot;REQUEST&quot; methods may be used to update the
   details of the event without rescheduling it, to update the
   &quot;PARTSTAT&quot; parameter of &quot;Attendees&quot;, or to reconfirm the event.

   this concept using the following workflow.  Any &quot;Attendee&quot; may
   delegate their right to participate in a calendar VEVENT to another

3.2.2.4. Changing the Organizer 



   The situation may arise where the &quot;Organizer&quot; of a VEVENT is no
   longer able to perform the &quot;Organizer&quot; role and abdicates without
   passing on the &quot;Organizer&quot; role to someone else.  When this occurs
   the &quot;Attendees&quot; of the VEVENT may use out-of-band mechanisms to
   communicate the situation and agree upon a new &quot;Organizer&quot;.  The new
   &quot;Organizer&quot; should then send out a new &quot;REQUEST&quot; with a modified
   version of the VEVENT in which the &quot;SEQUENCE&quot; number has been
   incremented and the &quot;ORGANIZER&quot; property has been changed to the new
   &quot;Organizer&quot;.

   another.  Using the &quot;SENT-BY&quot; parameter, a calendar user could send

3.2.2.6. Forwarding to An Uninvited CU 



   An &quot;Attendee&quot; invited to an event may invite another uninvited CU to
   the event.  The invited &quot;Attendee&quot; accomplishes this by forwarding
   the original &quot;REQUEST&quot; method to the uninvited CU.  The uninvited CU
   then replies to the &quot;Organizer&quot;.  The &quot;Organizer&quot; decides whether or
   not the uninvited CU is added to the attendee list.  If the
   &quot;Organizer&quot; decides not to add the uninvited CU no further action is
   required, however the &quot;Organizer&quot; MAY send the uninvited CU a
   &quot;CANCEL&quot; message.  If the &quot;Organizer&quot; decides to add an uninvited CU,
   a new &quot;ATTENDEE&quot; property is added for the uninvited CU with its
   property parameters set as the &quot;Organizer&quot; deems appropriate.  When
   forwarding a &quot;REQUEST&quot; to another CU, the forwarding &quot;Attendee&quot; MUST
   NOT make changes to the original message.

3.2.2.7. Updating Attendee Status 



   The &quot;Organizer&quot; of an event may also request updated status from one
   or more &quot;Attendees.  The &quot;Organizer&quot; sends a &quot;REQUEST&quot; method to the
   &quot;Attendee&quot; and sets the &quot;ATTENDEE;RSVP=TRUE&quot; property parameter.  The
   &quot;SEQUENCE&quot; property for the event is not changed from its previous
   value.  A recipient will determine that the only change in the
   &quot;REQUEST&quot; is that their &quot;RSVP&quot; property parameter indicates a request
   for updated status.  The recipient SHOULD respond with a &quot;REPLY&quot;
   method indicating their current status with respect to the &quot;REQUEST&quot;.

3.4.2.2. REQUEST for Update or Reconfirmation of a VTODO 



   The &quot;REQUEST&quot; method may be used to update or reconfirm a &quot;VTODO&quot;
   calendar component.  Reconfirmation is merely an update of &quot;Attendee&quot;
   completion status or overall &quot;VTODO&quot; calendar component status.

   An update to an existing &quot;VTODO&quot; calendar component does not involve
   changes to the start or due time or recurrence intervals, nor
   generally to the description for the &quot;VTODO&quot; calendar component.  If
   the recipient CUA of a &quot;REQUEST&quot; method finds that the &quot;UID&quot; property
   value already exists on the calendar and that the &quot;SEQUENCE&quot; property
   value in the &quot;REQUEST&quot; is the same as the value for the existing
   event, then the &quot;REQUEST&quot; method describes an update of the &quot;VTODO&quot;
   calendar component details, but not a rescheduling of the &quot;VTODO&quot;
   calendar component.

   The update &quot;REQUEST&quot; is the appropriate response to a &quot;REFRESH&quot;
   method sent from an &quot;Attendee&quot; to the &quot;Organizer&quot; of a &quot;VTODO&quot;
   calendar component.

   Unsolicited &quot;REQUEST&quot; methods MAY be sent by the &quot;Organizer&quot; of a
   &quot;VTODO&quot; calendar component.  The unsolicited &quot;REQUEST&quot; methods are
   used to update the details of the &quot;VTODO&quot; (without rescheduling it or
   updating the completion status of &quot;Attendees&quot;) or the &quot;VTODO&quot;
   calendar component itself (i.e., reconfirm the &quot;VTODO&quot;).

3.4.2.3. REQUEST for Delegating a VTODO 



   The &quot;REQUEST&quot; method is also used to delegate or reassign ownership
   of a &quot;VTODO&quot; calendar component to another &quot;Calendar User&quot;.  For
   example, it may be used to delegate an &quot;Attendee's&quot; role (i.e.
   &quot;chair&quot;, or &quot;participant&quot;) for a &quot;VTODO&quot; calendar component.  The
   &quot;REQUEST&quot; method is sent by one of the &quot;Attendees&quot; of an existing
   &quot;VTODO&quot; calendar component to some other individual.

3.4.2.4. REQUEST Forwarded To An Uninvited Calendar User 



   An &quot;Attendee&quot; assigned a &quot;VTODO&quot; calendar component may send the
   &quot;VTODO&quot; calendar component to another new CU, not previously
   associated with the &quot;VTODO&quot; calendar component.  The current
   &quot;Attendee&quot; assigned the &quot;VTODO&quot; calendar component does this by
   forwarding the original &quot;REQUEST&quot; method to the new CU.  The new CU
   can send a &quot;REPLY&quot; to the &quot;Organizer&quot; of the &quot;VTODO&quot; calendar
   component.  The reply contains an &quot;ATTENDEE&quot; property for the new CU.

   The &quot;Organizer&quot; ultimately decides whether or not the new CU becomes
   part of the to-do and is not obligated to do anything with a &quot;REPLY&quot;
   from a new (uninvited) CU.  If the &quot;Organizer&quot; does not want the new
   CU to be part of the to-do, the new &quot;ATTENDEE&quot; property is not added
   to the &quot;VTODO&quot; calendar component.  The &quot;Organizer&quot; MAY send the CU a
   &quot;CANCEL&quot; message to indicate that they will not be added to the to-
   do.  If the &quot;Organizer&quot; decides to add the new CU, the new &quot;ATTENDEE&quot;
   property is added to the &quot;VTODO&quot; calendar component.  Furthermore,
   the &quot;Organizer&quot; is free to change any &quot;ATTENDEE&quot; property parameter
   from the values supplied by the new CU to something the &quot;Organizer&quot;
   considers appropriate.

   &quot;SEQUENCE&quot; property for the &quot;VTODO&quot; is not changed from its previous

3.7. Implementation Considerations

3.7.1. Working With Recurrence Instances 



   iCalendar includes a recurrence grammar to represent recurring
   events.  The benefit of such a grammar is the ability to represent a
   number of events in a single object.  However, while this simplifies
   creation of a recurring event, meeting instances still need to be
   referenced.  For instance, an &quot;Attendee&quot; may decline the third
   instance of a recurring Friday event.  Similarly, the &quot;Organizer&quot; may
   change the time or location to a single instance of the recurring
   event.

   Since implementations may elect to store recurring events as either a
   single event object or a collection of discrete, related event
   objects, the protocol is designed so that each recurring instance may
   be both referenced and versioned.  Hence, implementations that choose
   to maintain per-instance properties (such as &quot;ATTENDEE&quot; property
   &quot;PARTSTAT&quot; parameter) may do so.  However, the protocol does not
   require per-instance recognition unless the instance itself must be
   renegotiated.

   The scenarios for recurrence instance referencing are listed below.
   For purposes of simplification a change to an event refers to a
   &quot;trigger property.&quot;  That is, a property that has a substantive
   effect on the meeting itself such as start time, location, due date
   (for &quot;VTODO&quot; calendar component components) and possibly description.

   &quot;Organizer&quot; initiated actions:

   o  &quot;Organizer&quot; deletes or changes a single instance of a recurring
      event
   o  &quot;Organizer&quot; makes changes that affect all future instances

Daboo Expires May 7, 2009 [Page 67]

Internet-Draft iTIP November 2008 



   o  &quot;Organizer&quot; makes changes that affect all previous instances
   o  &quot;Organizer&quot; deletes or modifies a previously changed instance

   &quot;Attendee&quot; initiated actions:

   o  &quot;Attendee&quot; changes status for a particular recurrence instance
   o  &quot;Attendee&quot; sends event-Counter for a particular recurrence
      instance

   An instance of a recurring event is assigned a unique identification,
   &quot;RECURRENCE-ID&quot; property, when that instance is renegotiated.
   Negotiation may be necessary when a substantive change to the event
   or to-do has been made (such as changing the start time, end time,
   due date or location).  The &quot;Organizer&quot; can identify a specific
   recurrence instance using the &quot;RECURRENCE-ID&quot; property.  The property
   value is equal to the date/time of the instance.  If the &quot;Organizer&quot;
   wishes to change the &quot;DTSTART&quot;, the original unmodified &quot;DTSTART&quot;
   value of the instance is used as the value &quot;RECURRENCE-ID&quot; property
   and the new &quot;DTSTART&quot; and &quot;DTEND&quot; values reflect the change.

3.7.2. Attendee Property Considerations 



   The &quot;ORGANIZER&quot; property is required on published events, to-dos, and
   journal entries for two reasons.  First, only the &quot;Organizer&quot; is
   allowed to update and redistribute an event or to-do component.  It
   follows that the &quot;ORGANIZER&quot; property MUST be present in the event,
   to-do, or journal entry component so that the CUA has a basis for
   authorizing an update.  Second, it is prudent to provide a point of
   contact for anyone who receives a published component in case of
   problems.

   There are valid [RFC2822] addresses that represent groups.  Sending
   email to such an address results in mail being sent to multiple
   recipients.  Such an address may be used as the value of an
   &quot;ATTENDEE&quot; property.  Thus, it is possible that the recipient of a
   &quot;REQUEST&quot; does not appear explicitly in the list.

   It is recommended that the general approach to finding a &quot;Calendar
   User&quot; in an attendee list be as follows:

   1.  Search for the &quot;Calendar User&quot; in the attendee list where
       &quot;CUTYPE=INDIVIDUAL&quot;
   2.  Failing (1) look for attendees where &quot;CUTYPE=GROUP&quot; or
       'CUTYPE=UNKNOWN&quot;.  The CUA then determines if the &quot;Calendar User&quot;
       is a member of one of these groups.  If so, the &quot;REPLY&quot; method
       sent to the &quot;Organizer&quot; MUST contain a new &quot;ATTENDEE&quot; property in
       which:

   3.
       *  the &quot;TYPE&quot; property parameter is set to INDIVIDUAL
       *  the &quot;MEMBER&quot; property parameter is set to the name of the

   When delegating an event request to another &quot;Calendar User&quot;, the

4.2.8. Forwarding an Event Request 



   The protocol does not prevent an &quot;Attendee&quot; from &quot;forwarding&quot; an
   &quot;VEVENT&quot; calendar component to other &quot;Calendar Users&quot;.  Forwarding
   differs from delegation in that the forwarded &quot;Calendar User&quot; (often
   referred to as a &quot;Party Crasher&quot;) does not replace the forwarding
   &quot;Calendar User&quot;.  Implementations are not required to add the &quot;Party
   Crasher&quot; to the &quot;Attendee&quot; list and hence there is no guarantee that
   a &quot;Party Crasher&quot; will receive additional updates to the event.  The
   forwarding &quot;Calendar User&quot; SHOULD NOT add the &quot;Party Crasher&quot; to the
   attendee list.  The &quot;Organizer&quot; MAY add the forwarded &quot;Calendar User&quot;
   to the attendee list.

4.7.2. Bad RECURRENCE-ID 



   Component instances are identified by the combination of &quot;UID&quot;,
   &quot;RECURRENCE-ID&quot;, and &quot;SEQUENCE&quot;.  When an &quot;Organizer&quot; sends an iTIP
   message to an &quot;Attendee&quot;, there are three cases in which an instance
   cannot be found.  They are:

   1.  The component with the referenced &quot;UID&quot; and &quot;RECURRENCE-ID&quot; has
       been found but the &quot;SEQUENCE&quot; number in the calendar store does
       not match that of the ITIP message.
   2.  The component with the referenced &quot;UID&quot; has been found, the
       &quot;SEQUENCE&quot; numbers match, but the &quot;RECURRENCE-ID&quot; cannot be
       found.
   3.  The &quot;UID&quot; and &quot;SEQUENCE&quot; numbers are found but the CUA does not
       support recurrences.

   In case (1), two things can happen.  If the &quot;SEQUENCE&quot; number of the
   &quot;Attendee's&quot; instance is larger than that in the &quot;Organizer's&quot;
   message then the &quot;Attendee&quot; is receiving an out-of-sequence message
   and MUST ignore it.  If the &quot;SEQUENCE&quot; number of the &quot;Attendee's&quot;
   instance is smaller, then the &quot;Organizer&quot; is sending out a newer
   version of the component and the &quot;Attendee's&quot; version needs to be
   updated.  Since one or more updates have been missed, the &quot;Attendee&quot;
   SHOULD send a &quot;REFRESH&quot; message to the &quot;Organizer&quot; to get an updated
   version of the event.

   In case (2), something has gone wrong.  Both the &quot;Organizer&quot; and the
   &quot;Attendee&quot; should have the same instances, but the &quot;Attendee&quot; does
   not have the referenced instance.  In this case the &quot;Attendee&quot; SHOULD
   send a &quot;REFRESH&quot; to the &quot;Organizer&quot; to get an updated version of the
   event.

5.2. Latency Issues 



   With a store-and-forward transport, it is possible for events to
   arrive out of sequence.  That is, a &quot;CANCEL&quot; method may be received
   prior to receiving the associated &quot;REQUEST&quot; for the calendar
   component.  This section discusses a few of these scenarios.

5.2.1. Cancellation of an Unknown Calendar Component. 



   When a &quot;CANCEL&quot; method is received before the original &quot;REQUEST&quot;
   method the calendar will be unable to correlate the &quot;UID&quot; property of
   the cancellation with an existing calendar component.  It is
   suggested that messages that can not be correlated that also contain
   non-zero sequence numbers be held and not discarded.  Implementations
   MAY age them out if no other messages arrive with the same &quot;UID&quot;
   property value and a lower sequence number.

5.2.2. Unexpected Reply from an Unknown Delegate 



   When an &quot;Attendee&quot; delegates an item to another CU they MUST send a
   &quot;REPLY&quot; method to the &quot;Organizer&quot; using the &quot;ATTENDEE&quot; properties to
   indicate that the request was delegated and to whom.  Hence, it is
   possible for an &quot;Organizer&quot; to receive an &quot;REPLY&quot; from a CU not
   listed as one of the original &quot;Attendees&quot;.  The resolution is left to
   the implementation but it is expected that the calendaring software
   will either accept the reply or hold it until the related &quot;REPLY&quot;
   method is received from the &quot;Delegator&quot;.  If the version of the
   &quot;REPLY&quot; method is out of date the &quot;Organizer&quot; SHOULD treat the
   message as a &quot;REFRESH&quot; message and update the delegate with the
   correct version provided that delegation to that delegate is
   acceptable.

   Under some conditions, a CUA may receive requests and replies with
   the same &quot;SEQUENCE&quot; property value.  The &quot;DTSTAMP&quot; property is
   utilized as a tie-breaker when two items with the same &quot;SEQUENCE&quot;
   property value are evaluated.
  1. Security Considerations 



   ITIP is an abstract transport protocol which will be bound to a real-
   time transport, a store-and-forward transport, and perhaps other
   transports.  The transport protocol will be responsible for providing
   facilities for authentication and encryption using standard Internet
   mechanisms that are mutually understood between the sender and
   receiver.

6.1. Security Threats

6.1.1. Spoofing the &8220;Organizer&8220; 



   In iTIP, the &quot;Organizer&quot; (or someone working on the &quot;Organizer's&quot;
   behalf) is the only person authorized to make changes to an existing
   &quot;VEVENT&quot;, &quot;VTODO&quot;, &quot;VJOURNAL&quot; calendar component and republish it or
   redistribute updates to the &quot;Attendees&quot;.  An iCalendar object that
   maliciously changes or cancels an existing &quot;VEVENT&quot;, &quot;VTODO&quot; or
   &quot;VJOURNAL&quot; calendar component may be constructed by someone other
   than the &quot;Organizer&quot; and republished or sent to the &quot;Attendees&quot;.

6.1.2. Spoofing the &8220;Attendee&8220; 



   In iTIP, an &quot;Attendee&quot; of a &quot;VEVENT&quot; or &quot;VTODO&quot; calendar component
   (or someone working on the &quot;Attendee's&quot; behalf) is the only person

Daboo Expires May 7, 2009 [Page 120]

Internet-Draft iTIP November 2008 



   authorized to update any parameter associated with their &quot;ATTENDEE&quot;
   property and send it to the &quot;Organizer&quot;.  An iCalendar object that
   maliciously changes the &quot;ATTENDEE&quot; parameters may be constructed by
   someone other than the real &quot;Attendee&quot; and sent to the &quot;Organizer&quot;.

6.1.3. Unauthorized Replacement of the Organizer 



   There will be circumstances when &quot;Attendees&quot; of an event or to-do
   decide, using out-of-band mechanisms, that the &quot;Organizer&quot; must be
   replaced.  When the new &quot;Organizer&quot; sends out the updated &quot;VEVENT&quot; or
   &quot;VTODO&quot; the &quot;Attendee's&quot; CUA will detect that the &quot;Organizer&quot; has
   been changed, but it has no way of knowing whether or not the change
   was mutually agreed upon.

6.1.4. Eavesdropping 



   The iCalendar object is constructed with human-readable clear text.
   Any information contained in an iCalendar object may be read and/or
   changed by unauthorized persons while the object is in transit.

6.1.5. Flooding a Calendar 



   Implementations could provide a means to automatically incorporate
   &quot;REQUEST&quot; methods into a calendar.  This presents the opportunity for
   a calendar to be flooded with requests, which effectively block all
   the calendar's free time.

6.1.6. Unauthorized REFRESH Requests 



   It is possible for an &quot;Organizer&quot; to receive a &quot;REFRESH&quot; request from
   someone who is not an &quot;Attendee&quot; of an event or to-do.  Only
   &quot;Attendee's&quot; of an event or to-do are authorized to receive replies
   to &quot;REFRESH&quot; requests.  Replying to such requests to anyone who is
   not an &quot;Attendee&quot; may be a security problem.

6.2. Recommendations 



   For an application where the information is sensitive or critical and
   the network is subject to a high probability of attack, iTIP
   transactions SHOULD be encrypted.  This may be accomplished using
   public key technology, specifically Security Multiparts for MIME
   [RFC1847] in the iTIP transport binding.  This helps mitigate the
   threats of spoofing, eavesdropping and malicious changes in transit.

6.2.1. Use of [RFC1847] to secure iTIP transactions 



   iTIP transport bindings MUST provide a mechanism based on Security
   Multiparts for MIME [RFC1847] to enable authentication of the

Daboo Expires May 7, 2009 [Page 121]

Internet-Draft iTIP November 2008 



   sender's identity, and privacy and integrity of the data being
   transmitted.  This allows the receiver of a signed iCalendar object
   to verify the identity of the sender.  This sender may then be
   correlated to an &quot;ATTENDEE&quot; property in the iCalendar object.  If the
   correlation is made and the sender is authorized to make the
   requested change or update then the operation may proceed.  It also
   allows the message to be encrypted to prevent unauthorized reading of
   the message contents in transit. iTIP transport binding documents
   describe this process in detail.

   Implementations MAY provide controls for users to disable this
   capability.

6.2.2. Implementation Controls 



   The threat of unauthorized replacement of the &quot;Organizer&quot; SHOULD be
   mitigated by a calendar system that uses this protocol by providing
   controls or alerts that make &quot;Calendar Users&quot; aware of such
   &quot;Organizer&quot; changes and allowing them to decide whether or not the
   request should be honored.

   The threat of flooding a calendar SHOULD be mitigated by a calendar
   system that uses this protocol by providing controls that may be used
   to limit the acceptable sources for iTIP transactions, and perhaps
   the size of messages and volume of traffic, by source.

   The threat of unauthorized &quot;REFRESH&quot; requests SHOULD be mitigated by
   a calendar system that uses this protocol by providing controls or
   alerts that allow &quot;Calendar User&quot; to decide whether or not the
   request should be honored.  An implementation MAY decide to maintain,
   for audit or historical purposes, &quot;Calendar Users&quot; who were part of
   an attendee list and who were subsequently uninvited.  Similar
   controls or alerts should be provided when a &quot;REFRESH&quot; request is
   received from these &quot;Calendar Users&quot; as well.
